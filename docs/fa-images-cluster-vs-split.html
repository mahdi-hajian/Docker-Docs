ایمیج‌های جدا در برابر docs-cluster

توضیح تخصصی: فرق ایمیج و کانتینر، کی ایمیج جدا لازم است، و چرا Compose و Kubernetes هر دو می‌توانند با یک ایمیج کار کنند.

۰) فرق ایمیج و کانتینر (دقیق)

مفهومچیست؟مثال
Imageبستهٔ فقط‌خواندنی روی دیسک/رجیستریonlyoffice/docs-cluster-de:latest
Container / Service / Podنمونهٔ در حال اجرا از روی یک ایمیجسرویس compose به نام docservice

یک ایمیج می‌تواند پایهٔ چند کانتینر با نقش متفاوت باشد. چند کانتینر جدا ≠ چند ایمیج جدا.

۱) سوءتفاهم رایج

جملهٔ اشتباه:

«در Docker Compose ایمیج‌های جدا لازم نیست، ولی در Kubernetes لازم است.»

جملهٔ درست:

نه Compose و نه Kubernetes از نظر فنی مجبور به ایمیج‌های جدا نیستند. فرق در قرارداد پیش‌فرض فایل اجرا است:

  • docker-compose.yml این ریپو از اول روی یک ایمیج چندنقشی (docs-cluster-*) نوشته شده است.
  • Helm chart رسمی به‌صورت پیش‌فرض (برای سازگاری عقب‌رو) نام‌های جدا می‌گذارد؛ ولی اجازه می‌دهد همه را به docs-cluster-* ببری.

پس موضوع «Compose در برابر K8s» نیست؛ موضوع این است: نقش پروسس چطور انتخاب می‌شود و tooling چه نام ایمیجی را می‌کشد.

۲) معماری منطقی (برای هر دو محیط یکی است)

تفکیک زیر، تفکیک نقش پروسس / workload است نه الزاماً تفکیک فایل ایمیج:

نقشمسئولیت
proxyNginx لبه، مسیریابی به سرویس‌های داخلی
docserviceهستهٔ ادیتور، API، همکاری هم‌زمان
converterتبدیل فرمت / CPU-bound؛ معمولاً بیشترین scale
adminpanelپنل مدیریت (editionهای تجاری)
Client → proxy → docservice ↘ converter (معمولاً از مسیر AMQP) Redis / PostgreSQL / RabbitMQ

Compose و Kubernetes هر دو همین توپولوژی را می‌سازند. تفاوت orchestration است، نه اجبار به تگ ایمیج جدا.

۳) واقعیت Dockerfile: ایمیج‌های جدا «مستعار نقش» هستند

تارگت اصلی runtime (docs → تگ docs-cluster-*) تقریباً همه فایل‌های لازم را دارد. سپس:

FROM docs AS proxy
ENTRYPOINT ["proxy-docker-entrypoint.sh"]

FROM docs AS docservice
CMD ["docservice"]

FROM docs AS converter
CMD ["converter"]

کامنت رسمی ریپو:

No additional files or configuration are added here; the image is identical to docs except for the default execution mode.
ایمیجمحتوا نسبت به clusterتفاوت واقعی
docs-cluster-*کاملentrypoint عمومی + CMD پیش‌فرض نقش پیش‌فرض
docs-proxy-*همان لایه‌هافقط ENTRYPOINT پیش‌فرض proxy
docs-docservice-*همان لایه‌هافقط CMD پیش‌فرض docservice
docs-converter-*همان لایه‌هافقط CMD پیش‌فرض converter

هیچ باینری مخصوص کوبر داخل ایمیج جدا نیست که در cluster نباشد. اندازهٔ یکسان (~۴.۹GB) در Desktop همین را نشان می‌دهد.

۴) Docker Compose این ریپو چه می‌کند؟

proxy:
  image: .../docs-cluster-de:...
  entrypoint: ["proxy-docker-entrypoint.sh"]

docservice:
  image: .../docs-cluster-de:...
  command: ["docservice"]

converter:
  image: .../docs-cluster-de:...
  command: ["converter"]

یعنی:

  1. همه از یک repository/tag می‌آیند.
  2. نقش با override سطح سرویس تعیین می‌شود.

نتیجه در Docker Desktop: docs-cluster-de در حال استفاده؛ سه تگ جدا unused.

Unused = «الان هیچ کانتینری به این tag اشاره نمی‌کند» — نه اینکه «روی کوبر حتماً لازم‌اند».

چرا این طراحی برای Compose منطقی است؟

  • یک منبع حقیقت برای لوکال
  • overrideی command در Compose طبیعی است
  • مدیریت تگ/دیسک ساده‌تر

آیا Compose می‌تواند از ایمیج جدا استفاده کند؟

بله؛ فقط این فایل طوری نوشته نشده. اگر image را به docs-docservice-de عوض کنی باز هم کار می‌کند چون محتوا یکی است.

۵) Kubernetes / Helm چه می‌کند؟

۵.۱ نیاز واقعی موتور Kubernetes

K8s به «سه‌تا فایل ایمیج» نیاز ندارد. نیاز واقعی‌اش:

  • چند Workload جدا برای lifecycle و scale جدا
  • هر Pod با command/args یا ENTRYPOINT مناسب
  • Service، PVC، SecurityContext، probe، HPA و …

کاملاً معتبر است که سه Deployment داشته باشی و هر سه image: docs-cluster-de:9.4.1 باشند و فقط args فرق کند — عین مدل Compose.

۵.۲ رفتار پیش‌فرض Helm chart رسمی

Chart Kubernetes-Docs به‌صورت تاریخی repository جدا دارد، مثلاً:

  • docs-docservice-de
  • docs-proxy-de
  • docs-converter-de

دلیل مستند خود chart:

kept for backward compatibility. You can use onlyoffice/docs-cluster-de instead.

یعنی پیش‌فرض Helm برای نشکستن آپگریدهای قدیمی نام جدا می‌گذارد؛ نه اینکه kubelet بدون ایمیج جدا نتواند کار کند.

۵.۳ چه وقتی «در کوبر سه‌تا لازم است» درست است؟

  1. values پیش‌فرض Helm را عوض نمی‌کنی → باید همان نام‌ها در رجیستری موجود باشند.
  2. سیاست سازمانی Image Policy روی نام repository جدا سخت‌گیر است.
  3. CI/CD برای هر سرویس digest جدا pin می‌کند (حتی اگر محتوا یکی باشد).
  4. نمی‌خواهی در PodSpec مقدار command را override کنی و به ENTRYPOINT/CMD پیش‌فرض ایمیج تکیه می‌کنی.

در همهٔ این حالت‌ها نیاز، تطابق با قرارداد tooling/عملیات است نه محدودیت معماری Docs.

۵.۴ چه وقتی در کوبر هم نیاز نیست؟

docservice.image.repository: registry/onlyoffice/docs-cluster-de
proxy.image.repository:      registry/onlyoffice/docs-cluster-de
converter.image.repository:  registry/onlyoffice/docs-cluster-de

با override بالا (و اطمینان از تنظیم صحیح entrypoint/args در chart)، فقط یک ایمیج در رجیستری کافی است.

۶) جدول تصمیم‌گیری

سؤالجواب
از نظر باینری سه‌تا ایمیج جدا لازم است؟خیر
از نظر معماری چند پروسس/Deployment جدا لازم است؟بله
چرا Compose این ریپو سه‌تا لازم ندارد؟نقش را با command/entrypoint روی cluster می‌دهد
چرا Helm انگار سه‌تا لازم دارد؟پیش‌فرض نام repository جداست (سازگاری عقب‌رو)
کی هر سه را push کنی؟وقتی values روی نام‌های جدا می‌ماند
کی فقط cluster کافی است؟Compose این ریپو؛ یا Helm با override به cluster
Unused یعنی چه؟الان هیچ کانتینری آن tag را اجرا نمی‌کند

۷) قیاس مفهومی

node dist/index.js proxy
node dist/index.js docservice
node dist/index.js converter
  • یک artifact + آرگومان‌های مختلف = مدل docs-cluster + Compose
  • سه تگ که فقط CMD پیش‌فرض فرق دارد = مدل docs-*-de + defaults Helm

از نظر runtime تقریباً معادل‌اند؛ دومی قرارداد عملیات/نام‌گذاری است.

۸) توصیه عملی

لوکال / Compose

  • روی docs-cluster-* بمان
  • تگ‌های جدا را در صورت کمبود فضا می‌توانی حذف کنی

Kubernetes

  1. ترجیح: یک ایمیج cluster در رجیستری + override همه repositoryها
  2. اگر Helm دست‌نخورده با defaults می‌ماند: هر سه تگ جدا را هم push کن
  3. همیشه جدا نگه دار: چند Deployment برای scale ≠ چند repository ایمیج (اختیاری)

۹) جمع‌بندی نهایی

Compose به ایمیج جدا «نیاز ندارد» چون نقش را override می‌کند. Kubernetes هم ذاتاً به ایمیج جدا «نیاز ندارد»؛ فقط Helm به‌صورت پیش‌فرض نام جدا می‌خواهد مگر values را عوض کنی.

نیاز واقعی همیشه: چند workload با نقش‌های متفاوت. نه لزوماً: چند فایل ایمیج متفاوت.
