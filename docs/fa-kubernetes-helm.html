Kubernetes / Helm — چه چیزی باید بالا بیاید؟

این صفحه می‌گوید روی کوبر چه workloadهایی لازم است، کدام اختیاری است، هر کدام چه می‌کند، و چرا لیست اجباری‌ها با Docker Compose فرق دارد.

ایمیج‌های رجیستری تو: 
repo.mohaymen.ir:3060/starrelease/onlyoffice/docs-cluster-de:none-root
repo.mohaymen.ir:3060/starrelease/onlyoffice/docs-proxy-de:none-root
repo.mohaymen.ir:3060/starrelease/onlyoffice/docs-docservice-de:none-root
repo.mohaymen.ir:3060/starrelease/onlyoffice/docs-converter-de:none-root
repo.mohaymen.ir:3060/starrelease/onlyoffice/docs-utils:none-root
repo.mohaymen.ir:3060/starrelease/onlyoffice/docs-metrics:none-root
repo.mohaymen.ir:3060/starrelease/onlyoffice/docs-postgresql:none-root

۱) اول این را بفهم: اجباری بودنِ «سرویس» ≠ اجباری بودنِ «ایمیج جدا»

  • روی K8s باید چند Deployment/Pod جدا داشته باشی (proxy، docservice، converter، …).
  • ولی لازم نیست برای هر کدام یک repository ایمیج جدا داشته باشی.
  • می‌توانی هر سه را از docs-cluster-de:none-root با args/command متفاوت اجرا کنی.
  • یا طبق پیش‌فرض Helm از docs-proxy-de / docs-docservice-de / docs-converter-de استفاده کنی (محتوا یکی است).

۲) چه چیزهایی روی Kubernetes اجباری است؟

الف) هسته Docs — بدون این‌ها ادیتور کار نمی‌کند

Workload وضعیت به چه درد می‌خورد؟ ایمیج پیشنهادی اگر نباشد؟
Deployment proxy + Service/Ingress اجباری ورودی ترافیک کاربران به کلاستر. Nginx لبه. docs-cluster-de:none-root یا docs-proxy-de:none-root کاربر به ادیتور نمی‌رسد.
Deployment docservice اجباری هستهٔ باز کردن سند، API، همکاری هم‌زمان. docs-cluster-de:none-root + args docservice
یا docs-docservice-de:none-root		 ادیتور کار نمی‌کند.
Deployment converter اجباری برای کار واقعی تبدیل فایل. معمولاً چند replica. docs-cluster-de:none-root + args converter
یا docs-converter-de:none-root		 تبدیل/باز کردن بعضی اسناد می‌شکند.
PVC / Shared storage برای cache فایل‌ها اجباری در معماری cluster فایل‌های موقت/کش مشترک بین podها (معمولاً زیر /var/lib/onlyoffice). — همکاری چندنودی و cache خراب می‌شود.

ب) وابستگی‌های خارجی — اجباری‌اند، ولی معمولاً «داخل chart Docs» نیستند

سرویس وضعیت به چه درد می‌خورد؟ چطور معمولاً می‌آوری؟ آیا باید ایمیج ONLYOFFICE باشد؟
PostgreSQL اجباری ولی خارجی دیتابیس وضعیت Docs Postgres سازمانی / Helm جدا / cloud DB + یک‌بار createdb.sql خیر. docs-postgresql فقط shortcut است.
Redis اجباری ولی خارجی کش و هماهنگی Redis معمولی / Sentinel / Cluster خیر — ایمیج رسمی Redis
RabbitMQ یا ActiveMQ اجباری ولی خارجی صف بین docservice و converter AMQP معمولی سازمان خیر

این سه تا از نظر Docs «لازم»‌اند، ولی در K8s برخلاف Compose معمولاً به‌عنوان سرویس داخل همان docker-compose.yml تعریف نمی‌شوند؛ از قبل در کلاستر موجود فرض می‌شوند یا با chart جدا نصب می‌شوند.

ج) اختیاری روی Kubernetes

Workload / جزء وضعیت به چه درد می‌خورد؟ ایمیج اگر نباشد؟
adminpanel اختیاری برای ادیتور UI مدیریت تنظیمات سرور docs-cluster-de:none-root + args adminpanel سند باز می‌شود؛ پنل ادمین نداری
metrics اختیاری StatsD / مانیتورینگ داخلی docs-metrics:none-root ادیتور کار می‌کند؛ متریک داخلی نداری
utils / observer scripts اختیاری / موقعیتی اسکریپت‌های کمکی balancer/observer در معماری پیشرفته K8s docs-utils:none-root برای استقرار ساده غالباً لازم نیست
example اختیاری اپ نمونه دمو docs-example روی production معمولاً نمی‌آید
docs-postgresql به‌عنوان Deployment اختیاری فقط اگر نخواهی Postgres سازمان را استفاده کنی docs-postgresql:none-root بهتر است Postgres استاندارد + Job برای SQL
ایمیج‌های docs-proxy-de / docs-docservice-de / docs-converter-de اختیاری فقط اگر Helm را با نام‌های پیش‌فرض جدا می‌خواهی همان‌ها با تگ none-root اگر همه را روی docs-cluster-de بگذاری لازم نیستند

۳) چرا لیست اجباری Compose با Kubernetes فرق دارد؟

این مهم‌ترین بخش است:

موضوع Docker Compose این ریپو Kubernetes / Helm چرا فرق دارد؟
هدف فایل/ابزار یک stack کامل «همه‌چیز با هم» برای لپ‌تاپ استقرار تولید روی کلاستر موجود Compose برای دمو/توسعه است؛ K8s برای عملیات سازمانی.
Postgres / Redis / RabbitMQ داخل همان docker-compose.yml تعریف و اجباری شده‌اند معمولاً خارجی‌اند (از قبل در کلاستر یا chart جدا) در سازمان اغلب DB/MQ مشترک از قبل وجود دارد؛ Helm Docs فرض می‌کند آدرس آن‌ها را می‌دهی، نه اینکه حتماً خودش Postgres بسازد.
سرویس‌های Docs (proxy/docservice/converter) اجباری اجباری این بخش در هر دو محیط یکی است — هستهٔ محصول.
تعداد ایمیج Docs عملاً ۱ ایمیج (cluster) برای چند سرویس ۱ ایمیج کافی است؛ یا ۳ نام جدا برای سازگاری Helm فرق در قرارداد نام‌گذاری است، نه در معماری لازم.
docs-postgresql در compose فعلی برای راحتی اجباری شده اختیاری؛ Postgres معمولی + SQL بهتر است در لوکال می‌خواهند یک دستور همه را بالا بیاورد؛ در کوبر نمی‌خواهند DB داخل همان اپ قفل شود.
utils در فایل هست ولی برای ادیتور لازم نیست فقط در سناریوهای خاص observer/balancer ابزار عملیات است، نه مسیر حیاتی باز کردن سند.
Ingress / Service پورت مستقیم 80:8888 روی proxy Service + Ingress/LoadBalancer اجباری برای دسترسی بیرونی مدل شبکه کلاستر فرق دارد.
PVC مشترک named volume ساده در معماری چندنودی خیلی مهم‌تر است چند pod روی چند node باید به دادهٔ مشترک برسند.

۴) پس «سرویس اجباری» در دو محیط یعنی چه؟

در Compose: هر چیزی که داخل همان فایل تعریف شده و بقیه به آن depends_on دارند، عملاً اجباریِ اجرای آن فایل است — حتی اگر از نظر معماری بشود جایش را با سرویس خارجی عوض کرد.

در Kubernetes: اجباری = چیزهایی که بدون آن‌ها Docs از کار می‌افتد (proxy/docservice/converter + DB + Redis + MQ + storage). ولی شکل آوردنشان آزادتر است: DB/Redis/MQ را خودت جدا می‌آوری.

۵) حداقل مجموعه پیشنهادی برای K8s تو

  1. اجباری Deploymentهای proxy + docservice + converter
  2. اجباری Postgres (معمولی) + یک‌بار createdb.sql
  3. اجباری Redis
  4. اجباری RabbitMQ/ActiveMQ
  5. اجباری Service/Ingress + PVC با fsGroup: 101
  6. اختیاری adminpanel / metrics / utils
  7. اختیاری ایمیج‌های split (docs-*-de) اگر همه را روی docs-cluster-de بگذاری

۶) نمونه values با ایمیج‌های تو

# همه نقش‌ها از یک ایمیج cluster
proxy:
  image:
    repository: repo.mohaymen.ir:3060/starrelease/onlyoffice/docs-cluster-de
    tag: none-root
docservice:
  image:
    repository: repo.mohaymen.ir:3060/starrelease/onlyoffice/docs-cluster-de
    tag: none-root
converter:
  image:
    repository: repo.mohaymen.ir:3060/starrelease/onlyoffice/docs-cluster-de
    tag: none-root

podSecurityContext:
  enabled: true
  fsGroup: 101

اگر Helm را دست‌نخورده با defaults بخواهی، به‌جای cluster از docs-proxy-de / docs-docservice-de / docs-converter-de با همان تگ none-root استفاده کن.
