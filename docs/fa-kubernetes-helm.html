راهنمای استفاده در Kubernetes و Helm

این راهنما بعد از ساخت ایمیج‌ها از ریپوی Docker-Docs، مسیر استقرار روی کلاستر را توضیح می‌دهد. Chart رسمی: ONLYOFFICE/Kubernetes-Docs

پیش‌فرض مهم

  • ایمیج‌های این ریپو برای دیپلوی cluster طراحی شده‌اند.
  • Clustering تجاری معمولاً با لایسنس Developer / Enterprise است.
  • کانتینرها به‌صورت پیش‌فرض با کاربر ds / UID 101 اجرا می‌شوند.

گام ۱ — بیلد و push ایمیج

# از ریپوی Docker-Docs
./build.sh          # یا .\build.ps1 روی ویندوز

# تگ برای رجیستری خودت
docker tag onlyoffice/docs-cluster-de:latest repo.example.com/onlyoffice/docs-cluster-de:9.4.1
docker tag onlyoffice/docs-utils:latest      repo.example.com/onlyoffice/docs-utils:9.4.1
docker tag onlyoffice/docs-metrics:latest    repo.example.com/onlyoffice/docs-metrics:9.4.1

docker push repo.example.com/onlyoffice/docs-cluster-de:9.4.1
docker push repo.example.com/onlyoffice/docs-utils:9.4.1
docker push repo.example.com/onlyoffice/docs-metrics:9.4.1

برای سازگاری با مقادیر پیش‌فرض قدیمی Helm می‌توانی ایمیج‌های docs-proxy-de / docs-docservice-de / docs-converter-de را هم push کنی؛ یا در values همه را به docs-cluster-de ببری.

گام ۲ — وابستگی‌ها

سرویستوضیح
PostgreSQL معمولی (≥ ۱۲.۹). لازم نیست ایمیج docs-postgresql باشد. یک‌بار createdb.sql را با Job/Init اجرا کن.
Redis برای کش و هماهنگی سشن‌ها
RabbitMQ (یا ActiveMQ) صف بین docservice و converter
Storage / PVC مسیر داده‌هایی مثل /var/lib/onlyoffice با مالکیت مناسب UID 101

گام ۳ — اجرای schema روی Postgres معمولی

نمونه مفهوم Job:

apiVersion: batch/v1
kind: Job
metadata:
  name: onlyoffice-db-init
spec:
  template:
    spec:
      restartPolicy: Never
      containers:
        - name: psql
          image: postgres:15
          env:
            - name: PGPASSWORD
              valueFrom:
                secretKeyRef:
                  name: onlyoffice-db
                  key: password
          command: ["bash","-lc"]
          args:
            - |
              psql -h $DB_HOST -U $DB_USER -d $DB_NAME -f /sql/createdb.sql
          volumeMounts:
            - name: sql
              mountPath: /sql
      volumes:
        - name: sql
          configMap:
            name: onlyoffice-createdb-sql

فایل SQL از پکیج Docs: مسیر تقریبی .../documentserver/server/schema/postgresql/createdb.sql

گام ۴ — مقادیر مهم Helm / securityContext

images:
  # مثال با رجیستری شخصی و ایمیج یکپارچه
  # در values واقعی هر کامپوننت repository خودش را دارد؛
  # می‌توانی همه را به docs-cluster-de تنظیم کنی.

podSecurityContext:
  enabled: true
  fsGroup: 101

docservice:
  containerSecurityContext:
    enabled: true
    runAsUser: 101
    runAsGroup: 101
    runAsNonRoot: true
  image:
    repository: repo.example.com/onlyoffice/docs-cluster-de
    tag: "9.4.1"

proxy:
  containerSecurityContext:
    enabled: true
    runAsUser: 101
    runAsGroup: 101
  image:
    repository: repo.example.com/onlyoffice/docs-cluster-de
    tag: "9.4.1"

converter:
  containerSecurityContext:
    enabled: true
    runAsUser: 101
    runAsGroup: 101
  image:
    repository: repo.example.com/onlyoffice/docs-cluster-de
    tag: "9.4.1"

پیش‌فرض chart ممکن است نام‌های جدا مثل docs-docservice-de باشد (backward compatibility). خود ONLYOFFICE گفته می‌توانی به‌جایشان docs-cluster-de بگذاری.

گام ۵ — لایسنس Developer/Enterprise

  • فایل خریداری‌شده: license.lic
  • معمولاً روی مسیر داده‌ای معادل /var/www/onlyoffice/Data/license.lic مونت می‌شود
  • بدون لایسنس، ایمیج -de ممکن است بالا بیاید ولی امکانات کامل/cluster محدود است

گام ۶ — نصب Helm (الگو)

helm repo add onlyoffice https://download.onlyoffice.com/charts/stable
helm repo update

helm upgrade --install docs onlyoffice/docs \
  -n onlyoffice --create-namespace \
  -f my-values.yaml

نام chart و repo را با نسخهٔ فعلی مستندات رسمی تطبیق بده؛ ممکن است با گذر زمان عوض شود.

چک سلامت بعد از دیپلوی

kubectl -n onlyoffice get pods
kubectl -n onlyoffice logs deploy/docs-docservice --tail=100
curl https://<your-ingress-host>/healthcheck
# انتظار: true

داخل pod باید کاربر غیر روت باشد (۱۰۱)، مگر اینکه securityContext را خاموش کرده باشی.

تفاوت با Docker Compose

موضوعComposeKubernetes
ورود ترافیک پورت ۸۰ روی proxy Ingress / Service + proxy Deployment
ایمیج Docs عمدتاً cluster cluster یا ایمیج‌های جدا (هر دو ممکن)
Postgres سرویس داخلی compose معمولاً خارجی/مدیریت‌شده + Job برای SQL
مقیاس --scale converter=N replicas / HPA روی Deploymentها
لایسنس cluster برای تست لوکال برای production چندنودی مهم‌تر است
اشتباه رایج: «Compose به ایمیج جدا نیاز ندارد ولی K8s نیاز دارد.»
درست: هر دو می‌توانند با docs-cluster کار کنند. Helm فقط به‌صورت پیش‌فرض نام جدا می‌گذارد (backward compatibility). توضیح کامل و تخصصی: fa-images-cluster-vs-split.html

چک‌لیست قبل از خرید/پروداکشن

  1. ایمیج با فیکس CRLF بیلد و push شده
  2. securityContext با ۱۰۱ فعال است و PVC قابل نوشتن است
  3. DB schema اعمال شده
  4. Redis و AMQP سالم‌اند
  5. JWT درست تنظیم شده
  6. برای محصول فروشی: مسیر Developer + لایسنس مناسب connections/cluster
